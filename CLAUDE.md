# barbar

A CEP panel (Svelte/TS, via bolt-cep) + an AEGP plugin (C++, `AegpDemo.aex`) for After Effects. The AEGP does all the work (hotkeys, popup, and running the ExtendScript); CEP is purely a settings UI over two JSON prefs files. There is no runtime link between the two - the prefs files *are* the contract.

## Architecture

- **AEGP side** (`AEGP/`): registers global hotkeys from the user's table ([HotkeyOverlay_Win32.cpp](AEGP/HotkeyOverlay_Win32.cpp)); they only fire if AE is the foreground process (PID check against `GetCurrentProcessId()` - cheap, since the plugin runs inside `AfterFX.exe` itself, no exe-path lookup needed). On trigger, shows a raylib popup ([HotkeyOverlay_Popup.cpp](AEGP/HotkeyOverlay_Popup.cpp)), rendered in-process on the hotkey thread (not a separate exe - `PopupExe/` is only a standalone dev runner for iterating on the UI without reloading AE):
  - A panel-sized window (the window *is* the slider panel, centered on the cursor, clamped onto the cursor's monitor); mouse input never comes from the window. The cursor is read globally per frame (`GetCursorPos`), modifiers via `GetAsyncKeyState`, and both mouse buttons via a `WH_MOUSE_LL` hook that also *eats* them so the commit/cancel click never reaches AE (the mouse guard, [EscGuard.cpp](AEGP/EscGuard.cpp)). An earlier design covered the whole monitor with a transparent window instead - that is unimplementable on hybrid-GPU laptops, see "Known gotchas".
  - A rounded slider panel that tracks the cursor continuously with no click/drag needed - drawn directly with raylib primitives (raygui is no longer used; an earlier version patched a `forceDragging` param into raygui's `GuiSliderPro`).
  - A left-click anywhere commits the slider value and dismisses the popup; a right-click cancels (restores the pre-drag keyframes), as does Esc.
  - The window is created fresh for every popup and fully torn down (`CloseWindow`) on dismiss, so raylib starts from a clean slate each call - see "Known gotchas" for the creation-time details that make this work.
- **Script execution**: the popup's actions land in AE as calls to the typed jsx functions (`applyEase`/`cancelEase`/... in [aeft.ts](src/jsx/aeft/aeft.ts)) run by the AEGP itself via `AEGP_ExecuteScript` - see [ScriptRunner.cpp](AEGP/ScriptRunner.cpp). `AEGP_ExecuteScript` is main-thread-only and the popup lives on the hotkey thread, so calls are queued and an AEGP **idle hook** drains them on AE's main thread; `AEGP_CauseIdleRoutinesToBeCalled()` (the one documented thread-safe utility call) pokes it awake. The compiled jsx bundle (`barbar-jsx.js`, shipped next to the .aex by `build.ps1`) is lazily `$.evalFile`'d on the first call of a session, creating the same `$["barbar"]` namespace object CEP's `evalTS` uses.
- **CEP side** (`src/js/`): two panels, both pure UI -
  - `main` - the hotkey editor + live-preview toggle.
  - `floating` - a draggable widget that closes itself on blur (`csi.closeExtension()` on window blur) - not reliable for anything that needs to persist.
  - Nothing breaks if no panel is ever opened; there is no server, no background panel, no port. (Both of those existed when the AEGP reached ExtendScript through CEP over a WebSocket - that whole layer is gone.)
- **Prefs files** (both in `%APPDATA%`, both written only by CEP - [prefs.ts](src/js/lib/prefs.ts)):
  - `barbar-hotkeys.json` - the hotkey table (`[{id,vkey,mods,fn},...]`, mods bitmask bit0=ctrl/bit1=shift/bit2=alt). The AEGP reads it at overlay-thread startup (hotkeys work from AE launch with no CEP running) and re-reads it when the overlay thread's directory watch (`FindFirstChangeNotification` on `%APPDATA%`) sees it change - the write is the whole notification, no ping crosses anywhere. A binding's `fn` is either `"Easing"` (the native popup) or the name of any exported `$["barbar"]` jsx function, called with no arguments - **adding a new tool = export a jsx function + add a table row, no C++ change**.
  - `barbar-settings.json` - `{"pollingEnabled":bool}`; the AEGP re-reads it at each popup open ([PopupActions.cpp](AEGP/PopupActions.cpp)) to decide whether to send live-preview ticks.

## Component internals

### AEGP threading model
Two threads matter. (1) **AE's main thread** runs the plugin ctor/hooks, including the ScriptRunner's idle hook - every ExtendScript call ultimately executes here, no matter who queued it. (2) **The overlay thread** (`ThreadProc` in [HotkeyOverlay_Win32.cpp](AEGP/HotkeyOverlay_Win32.cpp)) owns the hotkey message queue and, while a popup is up, runs the raylib loop inline. `RegisterHotKey`/`UnregisterHotKey` and `WM_HOTKEY` are all thread-affine, so everything hotkey-related is funneled here. Its loop is `MsgWaitForMultipleObjects` over the message queue *plus* a `FindFirstChangeNotification` handle watching `%APPDATA%` for hotkey-table edits (it can't watch a single file, so any change in that busy directory wakes it - the re-read + payload comparison makes unrelated churn a no-op). `StartHotkeyOverlay` blocks (â‰¤2s) on a `g_queueReady` event so a `WM_QUIT` racing in at startup can't be posted to a queue-less thread and dropped.

Because `RegisterHotKey` grabs a combo *system-wide* the instant it's registered, the table is only registered while AE is foreground: a 250ms `SetTimer` drives `ForegroundPollTick`, which registers/unregisters as focus crosses in/out (`IsAeForeground` = foreground window's PID == our PID; cheap because we *are* `AfterFX.exe`). Hotkeys are live from AE launch because `ThreadProc` seeds the table straight from `barbar-hotkeys.json` (`LoadHotkeyPayloadFromFile`) rather than waiting for CEP. On `WM_HOTKEY` it finds the monitor under the cursor (`MonitorFromPoint`, not the primary) and calls `RunHotkeyFunction`: `"Easing"` runs the popup in-process (see below); any other `fn` is sanitized to identifier chars and run as `$["barbar"].<fn>()` via the ScriptRunner.

### ScriptRunner mechanics ([ScriptRunner.cpp](AEGP/ScriptRunner.cpp))
- Suites (`AEGP_UtilitySuite6` for `AEGP_ExecuteScript`/`AEGP_CauseIdleRoutinesToBeCalled`, `AEGP_MemorySuite1` for the result handles) are acquired once in `StartScriptRunner` on the main thread and never released - suite acquisition isn't thread-safe, and the idle hook can't be unregistered, so the pointers must outlive everything.
- Every call is wrapped: lazy-load `barbar-jsx.js` if `$["barbar"]` is undefined, `try/catch` to an `"ERR:..."` result string (surfaced via `OutputDebugString`, never a dialog).
- `RunJsxAsync(call, coalesceKey)`: queued; entries with the same key collapse to the newest (keeping the oldest's queue position). Preview/commit/cancel all share the `"applyEase"` key - at most one is queued, newest wins - so a busy main thread applies the latest state once instead of replaying the drag, and a queued preview can never run after (or instead of) the commit/cancel that superseded it. Hold-Out uses no key: a queued preview should land first, in order, since hold only touches the outgoing interpolation type.
- `RunJsxSync(call, timeoutMs)`: condition-variable wait; a generation counter orphans a timed-out request so its late result is discarded rather than answering the next request.
- The idle hook refuses to re-enter itself (`g_executing`) - AE pumps idle routines from inside `AEGP_ExecuteScript`, and an unguarded re-entry runs the next queued script nested inside the current one (see Known gotchas).

### Popup interaction grammar ([HotkeyOverlay_Popup.cpp](AEGP/HotkeyOverlay_Popup.cpp))
`RunHotkeyFunction("Easing")` first asks AE whether anything is selected via the **blocking** `IsAnyKeyframeSelected()` (`RunJsxSync`, 1s cap), which runs jsx `beginEaseSession()`: besides answering, it snapshots every selected key's pre-drag state *before the popup window exists* - the one moment nothing else is queued or mid-write - and resets any leftover session. A timeout is treated as "selected" (err toward showing the popup; `applyEase`'s first tick re-snapshots as a fallback). Selected â†’ `RunPopupAtCursor`; not â†’ `RunNoSelectionToast` (a 1s fading "No keyframes are selected", dismissible early).

Inside `RunPopupAtCursor`'s loop, all state is read every frame so the drawing and the outgoing calls always agree - and all of it comes from *global* input reads, never from raylib's window events (the panel-sized window stops getting mouse events the instant the cursor leaves it, and it may never be focused at all - Windows' foreground-lock can deny it): the cursor via `GetCursorPos`, modifiers via `GetAsyncKeyState`, buttons via the mouse guard's latches (with raylib's events as the degraded fallback when the hook failed to install):
- **Slider value** = cursor-X mapped into the track, clamped 0â€“100.
- **Mode** from held modifiers: Alt â†’ `"in"`, Shift â†’ `"out"`, neither â†’ `"both"`. A mode flip counts as a change even when the value is still.
- **Ctrl = precision drag**: on Ctrl-down it anchors (value, mouseX); while held it moves at 1/30 speed relative to that anchor; on release, direct tracking resumes through a persistent X offset so the slider keeps the released value instead of jumping to the raw cursor position.
- **Icon**: value â‰¤ 0 draws the Linear glyph (no ease yet); otherwise the in/out/both easy-ease glyph for the current mode. Each state is a pre-colored SVG, not a runtime tint.
- **Hold-Out button** (right side, `Block.svg`): fires on release-while-hovered and is excluded from the generic "any click dismisses" check so its own press doesn't dismiss first.

Outgoing calls (all through [PopupActions.h](AEGP/PopupActions.h)): a **live preview** (`applyEase(value,mode,true)`) is polled every 0.25s only when value or mode changed - and dropped entirely when the user disabled polling (`PopupSessionBegin` re-reads the setting at each popup open). Exit paths: left-click â†’ `applyEase(value,mode,false)` (the commit); Hold-Out â†’ `setOutgoingHandleHold()`; **right-click or Esc** â†’ `cancelEase()` (Esc is read from the escape guard's hook, *not* from raylib - see the Esc gotcha). The mouse guard eats both buttons at the `WH_MOUSE_LL` level, so the commit/cancel click never reaches AE at all; only in the hook-failed fallback does the right-click path still wait for the button release before teardown (an orphan `WM_RBUTTONUP` in AE's queue becomes a `WM_CONTEXTMENU`). Dismissal is never focus-gated (see Known gotchas).

### The ease algorithm ([aeft.ts](src/jsx/aeft/aeft.ts))
`applyEase(value, mode, isPreview)` walks every selected property carrying selected keyframes (snapshotted into a plain array first, because `setInterpolationTypeAtKey` can deselect keys mid-walk). Per key:
- **Pre-drag snapshot** (`snapshotKeyOnce`, keyed by `layer|propName|propIndex|keyTime`) captures in/out interp types, temporal eases, and auto-bezier/continuous flags the *first* time a key is touched in a drag. Later polling ticks read from this snapshot, not from the previous tick's own output. `cancelEase` restores it; a commit clears the session so the next drag snapshots fresh. This state is module-level in whatever ExtendScript engine loaded the bundle and survives across calls (the AEGP loads the bundle once per AE session). Tangents are stored as plain `{speed,influence}` numbers, never live `KeyframeEase` host objects (those can go stale across calls and throw on restore). The restore is ordered bezier-both â†’ snapshot tangents â†’ original types â†’ flags, so no AE side effect (type-set tangent resets, `setTemporalEaseAtKey`'s force-to-bezier) can corrupt it; a key the restore had to skip (dead layer/keyframe) is appended to `%TEMP%/barbar-cancel.log` - check there first if a cancel ever "doesn't restore".
- **Interior-run detection** (`interiorRunFlags`): a key that a monotonic run passes straight through gets temporal auto-bezier (AE derives its tangents from neighbors â€” the smooth case) instead of the slider's ease, which belongs on the run's *ends*. Spatial props (motion-path-capable) use the dot product of incoming vs. outgoing motion (>0 = interior); everything else is per-dimension sign agreement with at least one dimension moving.
- **Mode** decides which side(s) of the ease the slider drives: `in` (Alt), `out` (Shift), or `both`; the untouched side is read back from the snapshot so in/out-only easing doesn't clobber it.

`isAnyKeyframeSelected` backs the selection query; `setOutgoingHandleHold` is the Hold-Out action.

## Build pipeline

- `pnpm build` = `tsc` + `vite build` (CEP bundle + the compiled jsx at `dist/cep/jsx/index.js`) -> `cd AEGP && build.ps1`.
- [build.ps1](AEGP/build.ps1) does, in order:
  1. CMake-vendors raylib via `FetchContent` (see [HotkeyOverlay/CMakeLists.txt](AEGP/HotkeyOverlay/CMakeLists.txt)).
  2. Copies headers/libs into `AEGP/Vendor/raylib/` (build *output*, gitignored and regenerated every build - don't hand-edit; `AEGP/Vendor/nanosvg/` is the exception: hand-vendored source, tracked).
  3. Ships the popup font/icons and `dist/cep/jsx/index.js` (as `barbar-jsx.js`) next to the .aex - the CEP build must have run first or it warns and the AEGP's script calls no-op until it ships.
  4. MSBuild builds `AegpDemo.vcxproj` straight into AE's plugin folder (`C:\Program Files\Adobe\Adobe After Effects 2026\Support Files\Plug-ins\AEGP\`).
- **Adobe SDK is not in the repo** (its EULA forbids redistribution): `AEGP/Headers/`, `AEGP/Util/`, and three files in `AEGP/Resources/` (`AE_General.r`, `Mach-O_prefix.h`, `PiPLtool.exe`) are gitignored and must be copied in from Adobe's free AE SDK download - see README "Adobe SDK". They exist in the local working tree, so local builds just work.
- `pnpm run build:aegp` runs just the AEGP step.
- `pnpm run reload` ([reload-ae.ps1](reload-ae.ps1)): kills `AfterFX.exe` -> `pnpm build` -> relaunches AE with the last-opened project. The "last opened project" lookup is a best-effort single-line regex match against AE's `Prefs.txt` MRU entry - it only works when that path is plain ASCII on one line. AE's prefs format wraps long/non-ASCII paths across multiple lines using an alternating literal-text / hex-encoded-UTF16BE quoted-segment encoding, which is not decoded here (attempted once, decoded garbage) - in that case AE just launches with no file.

**Standing rule: never run `pnpm reload`, `taskkill`, or `Stop-Process` against `AfterFX.exe` automatically. It runs elevated on this machine, so a non-elevated process gets Access Denied trying to kill it - this is a hard Windows privilege boundary, not something to route around. Tell the user a reload is needed and let them run it.**

## Known gotchas (already fixed, kept here so they don't get reintroduced)

- **MSBuild `OutDir`/`IntDir` trailing backslash**: a single trailing `\` right before the closing `"` in a quoted MSBuild arg escapes the quote instead of closing it, silently merging it with the next argument. `build.ps1` doubles any trailing backslash before passing `OutDir`/`IntDir` to MSBuild.
- **Stale CMake cache after moving the repo folder**: `HotkeyOverlay/build/CMakeCache.txt` hardcodes the absolute source path. If the repo directory ever gets moved/renamed, delete `AEGP/HotkeyOverlay/build/` before rebuilding, or CMake will refuse to reconfigure.
- **A monitor-covering transparent GL window is unimplementable on hybrid-GPU laptops - that's why the popup window is panel-sized**: DWM drops the alpha channel of GL framebuffers rendered by the discrete GPU when the display is driven by the iGPU. AfterFX.exe is bound to the NVIDIA GPU on this machine, so the popup's GL context inside AE always renders cross-adapter, and on the laptop's own (Intel-driven) panel the "transparent" fullscreen overlay rendered as a fullscreen *black* window - single-screen only, because the external monitor is NVIDIA-driven (that asymmetry is what made it look like a monitor-count bug). Nothing rescues the alpha: 1px overscan/undershoot, `WS_EX_LAYERED` at 255 and 254, `LWA_COLORKEY`, `DwmExtendFrameIntoClientArea`, the "disable fullscreen optimizations" compat flag - all tried, all still opaque; the identical window on the Intel GPU composites perfectly. The design consequence: the window is only as big as the panel (opaque content; the alpha bug can at worst blacken the sub-panel corner notches outside the rounding), and mouse capture comes from the mouse guard + global reads instead of window coverage. Don't reintroduce a monitor-sized transparent window, and when hunting popup rendering bugs, compare the GL `Vendor:` line in `%TEMP%\barbar-popup.log` between the AE run and a PopupExe run first - PopupExe defaults to the iGPU, AE doesn't, and that difference once burned days (force PopupExe onto the dGPU via `HKCU\Software\Microsoft\DirectX\UserGpuPreferences` â†’ `GpuPreference=2;` to reproduce AE's rendering path).
- **The popup window must still be created at its final size**: GLFW wires up the transparent framebuffer/DWM surface only at window creation - a 1x1 window resized afterward comes up opaque. This was originally misdiagnosed as "destroying/recreating the window breaks transparency", which led to a create-once/hide-show singleton that then needed workarounds of its own (`WindowShouldClose()`'s Esc-latch, `WM_DWMCOMPOSITIONCHANGED` nudges on re-show, foreground-lock denial of re-shown windows). The window is `InitWindow`/`CloseWindow` per popup - a clean raylib slate every call - and none of those workarounds exist anymore. Don't reintroduce the singleton. (`PopupExe`'s `recreate` mode runs two full window cycles in one process to test this path outside AE.)
- **Windows 11 power-throttles foreground-less processes hard enough to freeze the popup loop** ([popup_main.cpp](AEGP/PopupExe/popup_main.cpp)): the popup window is a no-activate toolwindow, so a process whose only window it is (PopupExe) counts as "background" - EcoQoS then coarsens timers so far that the `Sleep` inside raylib's `SetTargetFPS` pacing stretched to *seconds* (the loop visibly froze ~2s after open and woke on mouse input). PopupExe opts the whole process out via `SetProcessInformation(ProcessPowerThrottling)` (the timer-resolution control has no per-thread form); inside AfterFX.exe it can't engage - AE must be foreground for a hotkey to fire - and the overlay thread carries a per-thread `EXECUTION_SPEED` opt-out as insurance ([PopupWindowStyle.cpp](AEGP/PopupWindowStyle.cpp)).
- **Don't gate popup dismissal on `IsWindowFocused()`**: Windows' foreground-lock policy can silently deny focus to a window shown from a background thread, which once made the popup hide itself ~10 frames in (~166ms) regardless of user action. Dismissal is purely explicit: click anywhere, or Esc (via the escape guard - see next gotcha).
- **AE must never observe the popup's Esc press** ([EscGuard.cpp](AEGP/EscGuard.cpp)): AE aborts any running ExtendScript when it sees Esc - via its message queue (latched: it kills even a script that *starts* after the press) and via the live async key state polled mid-script - with "Unable to execute script at line N. Execution halted" at whatever line the abort happened to land on (the "random line number" alert). This applies regardless of who invokes the script - `AEGP_ExecuteScript` runs are exactly as killable as CEP `evalTS` runs were. Esc is the popup's cancel key, and cancel drives `cancelEase` in AE; preview ticks drive `applyEase` and are equally killable mid-run. Two half-fixes failed before the current design: sending `cancel` only after the key was physically released (AE still latched the keydown/auto-repeats that hit its queue once the popup window was torn down), and the queue-vs-async-state distinction generally (AE uses both). The fix is a `WH_KEYBOARD_LL` hook alive for the popup's lifetime that swallows Esc: a blocked low-level event reaches no window's queue *and* never enters `GetAsyncKeyState`, so every abort path is closed at once. Consequences baked into the popup loop: raylib never sees Esc while the hook is live (the loop reads `EscapeGuardTriggered()` instead of `WindowShouldClose`'s Esc latch), and on an Esc exit the transparent window stays up - pumping events so the hook keeps firing - until the key is physically released, only then is it torn down and `cancelEase` queued. If the hook fails to install everything degrades to the old raylib-Esc path with `WaitForEscapeReleased()` (async-key-state wait) as the remaining, imperfect, protection.
- **Idle-hook latency is real but bounded**: ExtendScript calls queue until AE's main thread goes idle - a render or modal dialog delays them, same as CEP's `evalScript` always did. That's why the popup's preview/commit/cancel coalesce (ScriptRunner keys) and why the selection query errs toward "selected" on its 1s timeout instead of failing closed.
- **ExtendScript parses nested ternaries left-associatively** - `a ? b : c ? d : e` runs as `(a ? b : c) ? d : e`, silently, in code that typechecks and behaves correctly in every non-ExtendScript engine (the CEP bundle included, since only the jsx bundle runs in AE). `auto ? "auto" : cont ? "continuous" : "broken"` therefore classified every auto-bezier key as `"continuous"` (`"auto"` is a truthy string) - which presented as *cancel restores continuous instead of auto* and, because the debug logging printed values derived through the same expression, spent days masquerading as a nondeterministic `keyTemporalAutoBezier` host bug (that getter is fine). Never nest ternaries in `src/jsx/**`; use if/else. When AE behavior contradicts what jsx code "obviously" does, suspect the ExtendScript *parse* of the shipped bundle before suspecting AE's API, and log raw inputs rather than derived values.
- **The ScriptRunner's idle hook has a reentrancy guard - it's load-bearing** ([ScriptRunner.cpp](AEGP/ScriptRunner.cpp) `g_executing`): AE services idle routines from inside `AEGP_ExecuteScript` (the same mid-script event pumping that lets Esc abort scripts), so when the popup thread queues a call and pokes `AEGP_CauseIdleRoutinesToBeCalled` while a script is executing, the hook re-enters. Without the guard the next queued script runs *nested between two statements* of the one still running - a preview tick's keyframe writes landing in the middle of another tick's snapshot reads. The guarded re-entry returns immediately; the outer drain loop picks the queue back up.
